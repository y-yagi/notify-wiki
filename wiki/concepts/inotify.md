# inotify

## Overview

inotify is a Linux kernel API for monitoring filesystem events on individual
files or directories. It was merged into Linux 2.6.13; the required glibc
interfaces landed in glibc 2.4 (with `IN_DONT_FOLLOW`, `IN_MASK_ADD`, and
`IN_ONLYDIR` following in glibc 2.5). It solves the problem of detecting file
changes (writes, creates, deletes, renames, attribute changes, opens/closes)
without polling, by having the application read structured event records off
a dedicated file descriptor. Since Linux 2.6.31, inotify's kernel-side event
delivery has been re-implemented on top of [[fsnotify]], the common backend
it now shares with [[dnotify]] and [[fanotify]]; this changed nothing about
the userspace-visible API.

## API / semantics

Four syscalls form the API:

- `inotify_init(2)` / `inotify_init1(2)` — create an inotify instance, returning
  a file descriptor. `inotify_init1(2)` adds a `flags` argument (e.g. `IN_NONBLOCK`).
- `inotify_add_watch(2)` — adds or modifies a "watch" on a pathname, specifying
  an event mask. Returns a "watch descriptor" (`wd`), a per-instance integer
  identifying that watch.
- `read(2)` — reads one or more `struct inotify_event` records from the fd. A
  blocking read blocks until at least one event is available.
- `inotify_rm_watch(2)` — removes a watch from the instance's watch list.

Closing all fds referencing an instance frees the instance and all its watches.

### Event structure

```c
struct inotify_event {
    int      wd;       /* Watch descriptor */
    uint32_t mask;      /* Mask describing event */
    uint32_t cookie;    /* Unique cookie associating related
                            events (for rename(2)) */
    uint32_t len;       /* Size of name field */
    char     name[];    /* Optional null-terminated name */
};
```

- `cookie` links the `IN_MOVED_FROM`/`IN_MOVED_TO` pair generated by a
  `rename(2)`; it's 0 for all other event types.
- `name` is present only for events on objects *inside* a watched directory
  (identifying which child); watches on individual files never populate it.
- Buffer-too-small behavior is kernel-version-dependent: pre-2.6.21 `read(2)`
  returns 0; since 2.6.21 it fails with `EINVAL`. A buffer of
  `sizeof(struct inotify_event) + NAME_MAX + 1` is guaranteed enough for at
  least one event.

### Watch model

Per-pathname watches (not per-fd, not recursive). Monitoring is **inode-based**:
watching an individual file catches activity via any link to that inode,
in any directory. Watching a directory catches:

- events marked with `*` (`IN_ATTRIB`, `IN_CLOSE_NOWRITE`, `IN_OPEN`) for both
  the directory itself and objects inside it, and
- events marked with `+` (`IN_ACCESS`, `IN_ATTRIB`, `IN_CLOSE_WRITE`, `IN_CREATE`,
  `IN_DELETE`, `IN_MODIFY`, `IN_MOVED_FROM`, `IN_MOVED_TO`) only for objects
  inside the directory, not the directory itself.

Directory watches are **not recursive** — subdirectories need their own
explicit watches, and there's an inherent race between creating a watch on a
new subdirectory and files already existing inside it by the time the watch
is added.

### Event/flag reference

Event bits (settable in `inotify_add_watch` mask, returned in `read()`'s mask):
`IN_ACCESS`, `IN_ATTRIB`, `IN_CLOSE_WRITE`, `IN_CLOSE_NOWRITE`, `IN_CREATE`,
`IN_DELETE`, `IN_DELETE_SELF`, `IN_MODIFY`, `IN_MOVE_SELF`, `IN_MOVED_FROM`,
`IN_MOVED_TO`, `IN_OPEN`. Convenience macros: `IN_ALL_EVENTS`, `IN_MOVE`
(= `IN_MOVED_FROM|IN_MOVED_TO`), `IN_CLOSE` (= `IN_CLOSE_WRITE|IN_CLOSE_NOWRITE`).

Watch-creation-only flags: `IN_DONT_FOLLOW` (since 2.6.15, don't dereference
symlinks), `IN_EXCL_UNLINK` (since 2.6.36, stop generating events for children
already unlinked from the watched directory), `IN_MASK_ADD` (OR into existing
mask instead of replacing), `IN_ONESHOT` (fire once then auto-remove),
`IN_ONLYDIR` (since 2.6.15, fail with `ENOTDIR` unless pathname is a directory
— race-free directory check), `IN_MASK_CREATE` (since Linux 4.18, fail with
`EEXIST` if a watch already exists on that inode, preventing accidental
clobbering when multiple paths alias the same inode).

Read-only flags (only ever appear in the returned mask, never settable):
`IN_IGNORED` (watch was removed, explicitly or automatically), `IN_ISDIR`
(subject is a directory), `IN_Q_OVERFLOW` (queue overflowed; `wd` is -1),
`IN_UNMOUNT` (containing filesystem was unmounted; followed by `IN_IGNORED`).

### /proc tunables

- `/proc/sys/fs/inotify/max_queued_events` — per-instance event queue cap
  (set at `inotify_init(2)` time); excess events are dropped but always
  trigger `IN_Q_OVERFLOW`.
- `/proc/sys/fs/inotify/max_user_instances` — instances per real UID.
- `/proc/sys/fs/inotify/max_user_watches` — watches per real UID.

## Limitations & gotchas

- **No PID/UID attribution** — can't tell which process triggered an event,
  including your own writes.
- **No network filesystem coverage** — only catches events triggered through
  the local filesystem API; remote/NFS-style changes need polling. Pseudo-filesystems
  like `/proc`, `/sys`, `/dev/pts` aren't monitorable either.
- **No `mmap`/`msync`/`munmap` coverage** — memory-mapped I/O doesn't generate events.
- **Filename identification is stale-prone** — by the time an app processes an
  event, the file may already be renamed or deleted.
- **App must maintain the wd→pathname cache itself**, and directory renames can
  invalidate multiple cached pathnames at once.
- **Not recursive** — watching a tree means watching every subdirectory individually,
  which is slow for large trees, plus the add-watch/scan race noted above.
  Per co-designer Robert Love, this was a deliberate design choice, not an
  oversight: recursive directory traversal isn't a first-class kernel
  operation on Unix-style filesystems (it's iterative listing + descend, same
  as userspace `find(1)`), so doing it inside the kernel — especially while
  holding locks — would be a too-long-running operation for no benefit over
  doing the same walk in userspace. The intent was for userspace libraries to
  implement the recursive walk themselves; the add-watch/scan race this
  creates is addressed by an algorithm Love and Trowbridge devised, referred
  to as "Love-Trowbridge".
- **Event coalescing** — consecutive identical events (same wd/mask/cookie/name)
  are coalesced into one if the older is still unread, so inotify cannot be
  used to reliably *count* events. (Before 2.6.25, the coalescing check was
  buggy — it compared against the oldest unread event instead of the most
  recent, see Platform notes.)
- **Queue overflow drops events** (`IN_Q_OVERFLOW`, wd = -1); apps must be able
  to rebuild their cache from scratch (e.g. tear down and recreate the fd and
  all watches).
- **Bind mounts over a watched directory generate no events**, and objects
  immediately under the new mount point aren't watched either, until unmounted again.
- **Rename-pair matching via `cookie` is inherently racy**: `IN_MOVED_FROM`/`IN_MOVED_TO`
  are usually consecutive but not guaranteed to be — other events can interleave,
  the pair isn't inserted atomically, and if the destination is outside any
  watched directory there may be no `IN_MOVED_TO` at all. Apps should tolerate
  fetching the pair across separate `read(2)` calls (with a short timeout) and
  treat unmatched `IN_MOVED_FROM` as a plain delete.
- **Watch descriptor recycling vs. pending events**: `wd` values are allocated
  incrementally (1 to `INT_MAX`) and can wrap around and be reused while older
  unread events for the previous holder of that `wd` are still queued — in
  principle causing an app to misattribute events to the wrong file. Documented
  as extremely unlikely to hit in practice and, as of the page's writing,
  intentionally left unfixed.

## Platform notes

- Merged into Linux 2.6.13; glibc support from 2.4 (2.5 for `IN_DONT_FOLLOW`,
  `IN_MASK_ADD`, `IN_ONLYDIR`).
- `IN_ONESHOT` was broken before 2.6.16.
- Signal-driven I/O (`O_ASYNC` via `fcntl(2)`) available for inotify fds since 2.6.25.
- Event coalescing logic was buggy before 2.6.25 (compared against oldest
  unread event instead of most recent).
- `IN_EXCL_UNLINK`, `IN_DONT_FOLLOW`, `IN_ONLYDIR` added 2.6.15/2.6.36 as noted above.
- `IN_MASK_CREATE` added in Linux 4.18.
- `fallocate(2)` generated no inotify events before Linux 3.19; since 3.19 it
  triggers `IN_MODIFY`.
- Since 2.6.36, `IN_ONESHOT` watches also emit `IN_IGNORED` when dropped
  (unintended side effect of other changes; originally they did not).
- Linux-only (`STANDARDS: Linux`).

## Related concepts

- `[[dnotify]]` — its predecessor, which it replaced in Linux 2.6.13.
- `[[fanotify]]` — sibling API with mount/filesystem-wide watching and access-
  permission interception; lacked create/delete/move events until Linux 5.1,
  before which inotify was the only option for those.
- `[[kqueue]]` — BSD/macOS equivalent, via the `EVFILT_VNODE` filter; a
  general kernel event queue rather than a file-notification-specific API,
  and requires an open fd per watched file rather than inotify's
  watch-descriptor model.
- `[[findfirstchangenotification]]` — Windows' oldest equivalent; unlike
  inotify's structured event records, it only signals that *some* change
  matching a filter occurred, with no event type or filename detail.
- `[[readdirectorychangesw]]` — Windows' closer analogue to inotify's event
  model: both deliver structured records (filename + change type) over a
  buffer/fd the app reads, with similar full-rescan-on-overflow requirements.
- `[[fsnotify]]` — the shared in-kernel backend inotify has been implemented
  on top of since Linux 2.6.31.
- `[[io_uring]]` — general Linux async I/O facility; conceivably usable to
  read an inotify fd asynchronously instead of blocking on `read(2)`, but no
  ingested source confirms this usage.
- [[recursive-watching]] — cross-cutting comparison of tree-watching support across all mechanisms/libraries in this wiki.
